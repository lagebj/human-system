# Domain Model

This skill covers overlapping domains that should not be treated as silos. The same underlying concerns travel through them: help, learning, responsibility, seams, judgment, capability, and system behavior.

## 1. Core human-systems domain

Primary concern:

How people, teams, organizations, and technical systems change each other through repeated behavior, pressure, relief, and responsibility.

Main resources:

- `resources/context/OPERATING_CONTEXT.md`
- `resources/source/core_stance/personal_alignment.md`
- `resources/source/core_stance/who_am_i.md`

Key concepts:

- Help changes the system it enters.
- Smoothness may be borrowed.
- Learning accumulates where uncertainty is carried.
- Responsibility moves through behavior before it moves on paper.
- Local optimization can weaken whole-system movement.

## 2. Operating-model / enablement domain

Primary concern:

How organizations actually change, how work moves across seams, and how help should strengthen capability rather than create dependency.

Main resources:

- `resources/source/field_patterns/trojan_horse_movement_pattern.md`
- `resources/source/core_stance/personal_alignment.md`
- `resources/source/core_stance/who_am_i.md`
- `resources/context/OPERATING_CONTEXT.md`

Key concepts:

- Modernization in place.
- Customer-owned capability.
- Temporary presence, permanent capability.
- Tooling follows context.
- Processes can be remedy, scaffold, scar tissue, or theatre.
- Governance should support judgment, not substitute for it.

## 3. Platform / EMS / technical strategy domain

Primary concern:

Building secure, automated, self-service test environment capability without letting technical implementation hide responsibility seams.

Main resources:

- `resources/context/OPERATING_CONTEXT.md`
- `resources/playbooks/technical-strategy-workflow.md`
- `resources/source/core_stance/personal_alignment.md`
- `resources/source/core_stance/who_am_i.md`

Key concepts:

- Ephemeral Azure test environments.
- Terraform, Ansible, Packer/image factory, GitLab, Artifactory, Vault.
- Role/base profiles.
- Vendor delivery decoupling.
- Platform vs app team ownership.
- Avoid forcing solutions when capability framing is more appropriate.

## 4. Youth football / Matchboard domain

Primary concern:

Creating humane, practical planning support for youth football that balances belonging, effort, development, appropriate challenge, and coach capacity.

Main resources:

- `resources/context/OPERATING_CONTEXT.md`
- `resources/playbooks/coaching-matchboard-workflow.md`
- `resources/playbooks/coding-agent-workflow.md`

Key concepts:

- Stable base groups with temporary movement.
- Effort, attendance, learning behavior, and game impact matter.
- Controlled differentiation with soft edges.
- Matchboard as planning/fairness tool.
- Pools as candidate structures alongside stable teams.
- Lineups, formations, rounds, fixtures, post-match reporting, involvement tracking.

## Cross-domain pattern

When the user asks about any of these domains, listen for the deeper seam:

- Where is responsibility carried?
- Where does learning accumulate?
- What has become easier in the moment but weaker later?
- What is being optimized locally while the whole becomes harder to move?
- Is the proposed help strengthening capability or absorbing uncertainty?
- Is the system learning to act, or learning to wait?
