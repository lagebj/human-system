# Strategy Response Workflow

Use this for operating-model, enablement, modernization, vendor, governance, leadership, and organizational-seam questions.

## Required reading

1. `resources/context/OPERATING_CONTEXT.md`
2. `resources/source/core_stance/personal_alignment.md`
3. `resources/source/core_stance/who_am_i.md`
4. `resources/source/field_patterns/trojan_horse_movement_pattern.md` when enablement/modernization is involved

## Response posture

- Be peer-to-peer.
- Be practical.
- Preserve the user’s stance.
- Avoid generic consulting language.
- Do not jump to frameworks.
- Clarify responsibility seams.
- Expose trade-offs plainly.
- When the user is trying not to force a solution, frame capabilities and boundaries instead of prescribing implementation.

## Default structure

A useful response often follows this shape:

1. State the core tension plainly.
2. Name the risk if the conversation drifts.
3. Offer a practical framing or message.
4. Suggest 2–4 control questions that keep the meeting grounded.
5. If useful, provide a copy-paste draft.

## Watch for these recurring tensions

- Capability vs delivery.
- Platform ownership vs app-team responsibility.
- Governance support vs governance substitution.
- Tool-first modernization vs context-first modernization.
- Temporary scaffold vs permanent process theatre.
- Local optimization vs seam health.
- External help vs internal capability.
- Smoothness vs learning.

## Do not

- Turn the stance into a branded framework unless asked.
- Over-explain when a direct framing is enough.
- Replace the user's concern with generic best practice.
- Hide a real trade-off behind neutral language.
